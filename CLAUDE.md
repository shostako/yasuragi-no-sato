# hpbuild - 介護施設ホームページ

## 概要
介護施設のコーポレートサイト。予約システム、ブログ更新機能、会員ログイン機能を備えた本格的なWebアプリケーション。

## 技術スタック
- **フレームワーク**: Next.js 16 (App Router)
- **バックエンド/DB**: Supabase (PostgreSQL, Auth)
- **スタイリング**: Tailwind CSS 4 + shadcn/ui
- **デプロイ**: Vercel（予定）

## デザイン方針
- **カラーパレット**: 茶系・ベージュ・オフホワイト（暖色系で安心感）
- **フォント**: 丸ゴシック系（読みやすさ・親しみやすさ）
- **トーン**: 清潔感 + 温かみ、信頼感・安心感重視

## ページ構成
- トップページ
- サービス紹介
- 会社概要
- お問い合わせ
- 採用情報
- ニュース・お知らせ

## 機能
| 機能 | 実装方法 |
|------|----------|
| 予約システム | Supabase DB + カレンダーUI |
| ブログ更新 | Supabase + 管理画面 |
| 会員ログイン | Supabase Auth |

## セッション開始時の必須アクション
**重要：新しいセッションでこのプロジェクトに入った際は、以下を最初に実行すること**

1. **PROGRESS.md を読む**: `PROGRESS.md` を開き、現在の状態と未完了タスクを把握
2. **次のアクション確認**: 「次セッションへの引き継ぎ」セクションを確認し、継続すべきタスクを認識
3. **ユーザーへの報告**: 認識した未完了タスク・Issueをユーザーに報告し、優先度を確認

**目的**: セッション間の引き継ぎを確実にし、前回の作業を途切れなく継続するため

## 作業ログの習慣
**重要：セッション終了前に必ず作業ログを記録すること**
- ログ保存先：`logs/yyyy-MM.md`
- その日の作業内容、技術的発見、教訓を記録

## ブラウザ自動検証ルール
Playwright MCPが有効な場合、以下のタイミングで自動的にブラウザ検証を行うこと：

### 検証タイミング
- フロントエンド実装完了時（新規ページ、コンポーネント追加）
- UI変更後の動作確認
- エラー発生時の状態確認

### 検証手順
1. ローカルサーバー起動（未起動の場合）
2. `playwright_navigate` でアクセス
3. 主要フローをテスト（入力、クリック、結果確認）
4. `playwright_screenshot` で状態記録
5. 問題発見時は修正→再検証のループ

### 自動検証の判断基準
- **する**: UI実装完了、スタイル変更、インタラクション追加
- **しない**: バックエンドのみの変更、設定ファイル変更、ドキュメント更新
